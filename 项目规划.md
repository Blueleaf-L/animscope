# 中国动画公司作品分析平台 v2 — 项目规划

## 一、项目概述

基于一份包含 **72 家中国动画公司、586 部作品** 的 Excel 数据，构建一个**前后端分离 + 数据库驱动** 的 Web 分析平台。v1 的经验和教训全部转化为本规划的硬性约束。

---

## 二、技术栈

| 层 | 技术 | 理由 |
|---|---|---|
| **数据库** | PostgreSQL 16 | 窗口函数、CTE、JSONB，分析型查询强 |
| **后端** | FastAPI + SQLAlchemy 2.0 (async) + asyncpg | 自动 Swagger 文档、Pydantic 校验、异步高性能 |
| **前端** | Vanilla HTML/CSS/JS + ECharts 5 (CDN) | 沿用 v1 验证过的模块化方案，零构建工具 |
| **Python 分析** | pandas + numpy + scipy | 数据导入 + 统计引擎 |
| **Python 可视化** | matplotlib / seaborn / plotly | 各司其职（见下） |
| **部署** | Docker Compose (FastAPI + PostgreSQL + Nginx) | 一键部署 |

### Python 可视化工具分工

| 工具 | 用途 | 交付形式 |
|---|---|---|
| **ECharts** | 前台交互式图表（环形图、折线、气泡、热力图） | JS 渲染，前端主导 |
| **matplotlib + seaborn** | 离线分析报告（PDF）、公司雷达图、箱线图、评级分布图 | 后端 API 按需生成 PNG/SVG |
| **plotly** | 行业仪表盘（多面板交互 HTML 片段嵌入页面） | `/api/v1/charts/industry-dashboard` 返回 HTML |

## 四、数据库设计

### 4.1 表结构

```sql
-- 动画公司表
CREATE TABLE companies (
    id            SERIAL PRIMARY KEY,
    name          VARCHAR(100) UNIQUE NOT NULL,
    type          VARCHAR(10) NOT NULL CHECK (type IN ('2D', '3D', '三渲二')),
    created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- 作品表
CREATE TABLE works (
    id            SERIAL PRIMARY KEY,
    company_id    INTEGER NOT NULL REFERENCES companies(id) ON DELETE CASCADE,
    name          VARCHAR(200) NOT NULL,
    year          INTEGER,
    rating_raw    VARCHAR(50),
    rating_label  VARCHAR(50),
    rating_score  NUMERIC(4,2) CHECK (rating_score BETWEEN -2.0 AND 6.0),
    quantity      INTEGER DEFAULT 1,
    created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

### 4.2 索引

```sql
CREATE INDEX idx_works_company ON works (company_id);
CREATE INDEX idx_works_year ON works (year);
CREATE INDEX idx_works_rating_score ON works (rating_score DESC);
CREATE INDEX idx_works_rating_label ON works (rating_label);
CREATE INDEX idx_companies_type ON companies (type);
CREATE INDEX idx_companies_name ON companies (name);
```

### 4.3 设计原则

- **2 张表**，不冗余。`works_count` 通过 `COUNT(*)` 子查询实时获取
- 所有分析数据由后端 `analysis_service.py` 实时从 DB 查询 + numpy/scipy 计算，**不做预聚合存储**
- 索引覆盖 90% 的查询模式。586 条数据规模下，无需物化视图

## 六、前端架构

### 6.1 模块分层

```
config.js      ← 全局配置 (API_BASE, getColors() 动态读取 CSS 变量)
api.js         ← API 调用层，统一 fetch 封装 + 超时控制 + 重试 + 加载态/错误态（详见 6.5）
analytics.js   ← 纯函数计算 (仅客户端特有逻辑, 不重复后端计算)
charts.js      ← ECharts 渲染 (颜色从 CONFIG.getColors() 读取, 不硬编码)
views.js       ← DOM 渲染 + 动画
router.js      ← Hash 路由 (≤150行)
pages/*.js     ← 页面模块 (每个 ≤300行)
app.js         ← 入口: 初始化 → 骨架屏 → API预加载 → 渲染
```

### 6.2 编码规范

- 单文件 ≤ 300 行，超过即拆分
- ECharts 颜色统一从 `CONFIG.getColors()` 获取，禁止硬编码色值
- 所有 `innerHTML` 数据插入点必须经过 `escapeHtml()` 转义
- CSS 变量为颜色唯一真相源，JS 通过 `getComputedStyle` 读取

### 6.3 config.js 核心设计

```js
const CONFIG = {
  API_BASE: '/api/v1',
  PAGE_SIZE: 20,
  getColors() {
    const s = getComputedStyle(document.documentElement);
    return {
      rating: {
        年度推荐: s.getPropertyValue('--rating-best').trim(),
        佳作: s.getPropertyValue('--rating-great').trim(),
        还行: s.getPropertyValue('--rating-good').trim(),
        能看: s.getPropertyValue('--rating-ok').trim(),
        不明: s.getPropertyValue('--rating-unknown').trim(),
        拉了: s.getPropertyValue('--rating-bad').trim(),
        史: s.getPropertyValue('--rating-trash').trim(),
      },
      palette: ['#5b8def', '#f06b4a', '#3cc9a6', '#7c5ce0', '#f5a623', '#e8405d'],
    };
  },
};
```

> `getColors()` 每次调用都动态读取 CSS 变量，确保暗色模式切换后 ECharts 图表颜色实时更新。

### 6.4 api.js 核心设计

```js
const API = {
  BASE: '/api/v1',
  TIMEOUT: 8000,  // 8 秒超时

  async request(path, opts = {}) {
    const controller = new AbortController();
    const timer = setTimeout(() => controller.abort(), this.TIMEOUT);

    const raw = opts.raw;        // raw=true 返回 Response 对象（用于二进制/图片）
    delete opts.raw;

    let res;
    try {
      res = await fetch(this.BASE + path, {
        ...opts,
        signal: controller.signal,
        headers: { 'Accept': opts.raw ? '*/*' : 'application/json', ...opts.headers },
      });
    } catch (err) {
      if (err.name === 'AbortError') throw new ApiError('请求超时，请检查网络后重试', 'TIMEOUT');
      throw new ApiError('网络连接失败', 'NETWORK');
    } finally {
      clearTimeout(timer);
    }

    if (!res.ok) {
      // 尝试解析错误详情
      let detail = `服务器错误 (${res.status})`;
      try {
        const body = await res.json();
        detail = body.detail || detail;
      } catch (_) { /* 非 JSON 响应 */ }
      throw new ApiError(detail, `HTTP_${res.status}`);
    }

    return raw ? res : res.json();
  },

  // 便捷方法
  get(path)               { return this.request(path); },
  getRaw(path)            { return this.request(path, { raw: true }); },   // 图表 API、PDF 下载用
  post(path, data) { return this.request(path, { method:'POST', body:JSON.stringify(data), headers:{'Content-Type':'application/json'} }); },
};
```

**错误处理流程**：
- `ApiError` 自定义类，携带 `code`（TIMEOUT / NETWORK / HTTP_500 等）和用户可读的 `message`
- 前端统一捕获：网络超时 → 全局 Toast "请求超时，正在重试..." → 自动重试 1 次；HTTP 错误 → 根据 status 显示对应提示
- 重试策略：仅对 GET 请求 + 超时/NETWORK 错误自动重试 1 次，POST/PUT 不重试（防重复提交）
- 加载态：每个 API 调用暴露 `{ loading, error, data }` 三态，页面组件根据状态渲染骨架屏/错误提示/正常内容

## 八、移动端与响应式策略

### 8.1 断点定义

沿用 v1 的三级断点体系：

| 断点 | 宽度 | 目标设备 | 核心策略 |
|------|------|---------|---------|
| 桌面 | ≥ 1024px | PC / 笔记本 | 全部图表正常渲染，多列布局 |
| 平板 | 768px – 1023px | iPad / 大屏手机横屏 | 图表区从两列变为单列，表格横向可滚动 |
| 手机 | < 768px | 手机竖屏 | 图表降级渲染，导航收缩为汉堡菜单 |

### 8.2 图表降级策略（复用 v1 方案）

| 图表类型 | 桌面 (≥1024px) | 平板 (768-1023px) | 手机 (<768px) |
|---------|-----------------|-------------------|---------------|
| 环形图 | 400×400 | 300×300 | **降级为柱状图** |
| 玫瑰图 | 400×400 | 300×300 | **降级为水平柱状图** |
| 气泡散点图 | 100%宽 | 100%宽 | **降级为表格** |
| 热力图 | 100%宽 | 缩小单元格 | X轴标签旋转45° |
| 公司表格 | 全列渲染 | 横向可滚动 | 压缩为 4 列，点击展开 |
| 作品时间线（横向） | ECharts 散点图 | 字体缩小 | **降级为纵向 DOM 列表** |
| 卡片网格 | 4 列 | 3→2 列 | 单列 |
| 雷达图 (SVG) | 400×400 | 300×300 | CSS 缩放 `max-width:100%` |

### 8.3 实现方式

- `charts.js` 每个 render 函数内部检查 `window.innerWidth < 768` 切换降级分支
- 图表初始化时绑定 `window.resize` 事件（200ms 防抖），跨越断点时重渲染
- 导航栏移动端使用纯 CSS checkbox hack 实现汉堡菜单（复用 v1 方案）
- 根字号移动端缩小：`html { font-size: 14px }` @ `<768px`
- 暗色模式 + 移动端组合：CSS 变量体系天然适配，无需额外处理

## 九、核心服务设计

### 9.1 analysis_service.py

```
calc_overview(db)          → 首页概览数据
calc_company_strength(db)  → Z-Score 综合实力排名
calc_risk_alerts(db)       → 风险预警 (trend_slope + volatility)
calc_track_benchmarks(db)  → 赛道基准线 (P65 per type)
calc_compare(db, ids)      → 多公司对比指标
calc_diff(db, a, b)        → Cohen's d + 维度对决 + σ 波动
calc_heatmap(db, top_n)    → 公司×年份评分矩阵
```

**原则**：一次 DB 查询获取原始数据 → numpy 数组 → 计算 → Pydantic 序列化。不存中间结果。

### 9.2 chart_service.py

```
chart_rating_dist(db, theme)     → (bytes, "image/png")
chart_company_radar(db, id, theme) → (bytes, "image/svg+xml")
chart_heatmap(db, top_n, theme)  → (bytes, "image/png")
chart_boxplot(db, type_filter)    → (bytes, "image/png")
chart_compare_radar(db, ids, theme) → (bytes, "image/svg+xml")
chart_dashboard_html(db)          → str  (plotly HTML 片段)
chart_report_pdf(db)              → (bytes, "application/pdf")
```

- matplotlib/seaborn 图表返回 `StreamingResponse(bytes, media_type)`，用于 PDF 报告和按需下载
- plotly 仪表盘返回 `HTMLResponse(str)`，唯一嵌入网页的 Python 图表
- 网页交互式图表继续由 ECharts 主导，Python 图表不替代 ECharts（吸取 v1 教训）
- report 页核心功能：**📥 下载 PDF 分析报告** + plotly 仪表盘预览
- 暗色模式通过 `?theme=dark` 参数控制配色

```
测试金字塔 —— 集成测试: 所有 API 端点 + 真实测试数据库
        \
         \ matplotlib 图表: 只测"不抛异常"，不做像素对比
```

## 十二、部署方案

